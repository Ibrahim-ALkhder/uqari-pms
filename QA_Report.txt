  1. QA STRATEGY & METHODOLOGY
================================================================================

  1.1 Testing Pyramid Approach

  The project follows a three-tier testing pyramid, prioritizing speed and
  isolation at the base and full-system validation at the apex.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    ACCEPTANCE TESTS (Manual + E2E)                     │
  │   ● Full user workflows: login → create tenant → add contract →       │
  │     generate invoice → record payment                                  │
  │   ● Arabic UI validation: message correctness, RTL rendering           │
  │   ● Cron job simulation: manual trigger of autoGenerateInvoices()     │
  │   ● Cross-browser: Chrome, Edge (RTL support)                        │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                    INTEGRATION TESTS (API-Level)                       │
  │   ● HTTP status codes: 200, 201, 400, 401, 404, 422                   │
  │   ● Auth middleware: valid JWT, expired JWT, missing header            │
  │   ● Database transactions: atomicity of payment + invoice update       │
  │   ● CORS validation: allowed (3000, 5173) vs disallowed origins        │
  │   ● Proxy verification: Vite → Express request routing                 │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                    UNIT TESTS (Function-Level)                         │
  │   ● Input validation: email regex, password length, date format        │
  │   ● Business logic: balance calculation, credit detection,             │
  │     invoice status transitions (Unpaid → Partial → Paid)               │
  │   ● Cron helpers: billing day comparison, due day clamping,            │
  │     invoice number sequence generation (INV-YYYY-MM-SSS)               │
  │   ● Arabic message assertions: correctness of 20+ error/success strings│
  └─────────────────────────────────────────────────────────────────────────┘

  1.2 Testing Tools & Frameworks

  | Tier               | Tool             | Scope                              |
  |--------------------|------------------|------------------------------------|
  | Unit               | Vitest           | Pure logic, validation functions   |
  | Integration        | Supertest + Vitest | Express routes, DB operations   |
  | Acceptance (BE)    | PowerShell curl  | Manual API verification            |
  | Acceptance (FE)    | Browser DevTools | React component rendering, network |

  1.3 Test Data Strategy

  - A dedicated seed script (server/seed.js) populates the database with
    1 landlord, 3 properties, 15 units, 11 tenants, ~50 invoices, and
    35+ payments with known values.
  - Tests run against the seeded database. After each destructive test
    (e.g., property deletion), the seed is re-run to restore baseline.
  - The database file is located at: server/data/pms_database.db

  3.1 Authentication & Security
  ─────────────────────────────────────────────────────────────────────────────

  TC-AUTH-01: Successful Login with Valid Credentials
  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that a registered user can authenticate using correct email and
    password, and receives a JWT token valid for 7 days.
  Precondition:
    Database seeded with user ahmed@example.com / secret123
  Test Steps:
    1. Send POST /api/auth/login with body { "email": "ahmed@example.com",
       "password": "secret123" }
    2. Verify response status is 200
    3. Verify response body contains "success": true
    4. Verify response body contains "data.token" with a non-empty string
    5. Decode token and verify payload contains userId, role, email
    6. Verify token expiration is 7 days from issuance
  Expected Result:
    HTTP 200. Token returned and valid.
  Actual Result:   PASS ✓
  Notes (ملاحظات):
    تم تسجيل الدخول بنجاح. التوكن صالح لمدة 7 أيام ويحتوي على معرف المستخدم
    والبريد الإلكتروني.

  TC-AUTH-02: Login with Invalid Password
  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that login with an incorrect password returns a 401 error with
    an Arabic error message.
  Test Steps:
    1. Send POST /api/auth/login with body { "email": "ahmed@example.com",
       "password": "wrongpassword" }
    2. Verify response status is 401
    3. Verify response body contains "success": false
    4. Verify response message is in Arabic
  Expected Result:
    HTTP 401. Arabic error message: "البريد الإلكتروني أو كلمة المرور غير صحيحة."
  Actual Result:   PASS ✓

  TC-AUTH-03: Access Protected Route Without Token
  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that accessing any /api/* endpoint without an Authorization header
    returns HTTP 401 with a proper early-return pattern (no next() call).
  Test Steps:
    1. Send GET /api/properties with no Authorization header
    2. Verify response status is 401
    3. Verify response body contains "success": false
    4. Verify Arabic message: "يرجى تسجيل الدخول أولاً."
  Expected Result:
    HTTP 401. No fallthrough to next().
  Actual Result:   PASS ✓

  TC-AUTH-04: Access Protected Route with Expired Token
  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that an expired or malformed JWT is rejected with 401.
  Test Steps:
    1. Send GET /api/dashboard/summary with Authorization:
       "Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJleHAiOjE1MDAwMDB9."
    2. Verify response status is 401
  Expected Result:
    HTTP 401.
  Actual Result:   PASS ✓

  TC-AUTH-05: PIN Verification (4-digit)
  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that a registered PIN can be verified and that an incorrect PIN
    is rejected.
  Test Steps:
    1. Login to obtain token
    2. Send POST /api/auth/verify-pin with { "pin": "1234" }
    3. Verify 200 on correct PIN
    4. Send POST /api/auth/verify-pin with { "pin": "0000" }
    5. Verify 401 on incorrect PIN
  Expected Result:
    Correct PIN → 200. Incorrect PIN → 401.
  Actual Result:   PASS ✓

  ─────────────────────────────────────────────────────────────────────────────
  Description:
    Verify that deleting a property requires the Arabic word "حذف" (delete)
    in the request body. Any other word is rejected.
  Test Steps:
    1. Attempt DELETE /api/properties/:id with body
       { "confirmationWord": "delete" } (English)
    2. Verify 400 with Arabic error message
    3. Attempt DELETE /api/properties/:id with body
       { "confirmationWord": "حذف" } (correct Arabic)
    4. Verify 200
  Precondition:
    Property must have no active contracts or unpaid invoices.
  Expected Result:
    English word → 400. Arabic "حذف" → 200.
  Actual Result:   PASS ✓
  Notes (ملاحظات):
    خاصية الأمان هذه مصممة خصيصاً لكبار السن لمنع الحذف العرضي.
    المستخدم يجب أن يكتب كلمة "حذف" بالعربية لتأكيد العملية.

  5.1 Known Issues

  ID:   KNW-001
  Type: Cosmetic
  Area: Frontend / Console
  Desc: React StrictMode in development causes double-mount of useEffect hooks.
        This results in duplicate API calls appearing in the Network tab.
        The extra calls return identical data and do not affect behaviour.
  Impact: Low (development only, StrictMode is stripped from production builds)
  Workaround: None required; expected React 19 behaviour.

  ID:   KNW-002
  Type: Limitation
  Area: Financial Cron Engine
  Desc: If a short month (e.g., February) has fewer days than the configured
        billingDay (e.g., billingDay = 31), the cron job will never trigger
        for that month because today !== billingDay never matches.
  Impact: Low (billingDay default is 1; most users set 1–5).
  Workaround: Implement day clamping: if billingDay > daysInMonth, use
              daysInMonth as the effective billing day for that month.
  Reference: financialCron.js line 58 already clamps dueDay with
             `const safeDueDay = Math.min(dueDay, daysInMonth)` — the same
             logic should be applied to billingDay comparison.

  ID:   KNW-003
  Type: Limitation
  Area: Payment Logic
  Desc: The overpayment credit is stored as a text note in the invoice.notes
        field. There is no automated mechanism to apply this credit to future
        invoices — the landlord must manually track it.
  Impact: Medium (manual workaround exists; credit is visible in notes)
  Recommendation: Add a cron job step that, during autoGenerateInvoices(),
                  checks for pending credits on each tenant and automatically
                  reduces the new invoice amount by the available credit.

  ID:   KNW-004
  Type: Missing Feature
  Area: Undo Payment
  Desc: The frontend Dashboard displays an "Undo" button for payments made
        within the last 24 hours, but the backend does not implement a
        DELETE /api/payments/:id endpoint. The button triggers a confirmation
        dialog and mock toast, but no actual rollback occurs.
  Impact: Low (visual feature only — displayed but non-functional)
  Recommendation: Implement the backend endpoint that deletes a payment,
                  recalculates invoice status, and restores previous balance.
