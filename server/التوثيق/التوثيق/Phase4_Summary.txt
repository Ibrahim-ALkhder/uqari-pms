================================================================================
  AUTHENTICATION & MIDDLEWARE (middleware/authMiddleware.js)
================================================================================

- JWT-based authentication with bcrypt (salt rounds 12)
- Optional 4-digit PIN verification
- Three middleware functions:
  authenticate       - Requires valid JWT; early return pattern for all errors
  optionalAuthenticate - Attaches user if token present, continues otherwise
  authorize          - Role-based access control
- PIN hashing with bcrypt, stored alongside password hash
- All error paths return 401 with res.status().json(); return; (no next())

Auth Endpoints (5):
  POST /api/auth/register     - Create account (name, email, password, pin)
  POST /api/auth/login        - Login (returns JWT)
  POST /api/auth/verify-pin   - Verify 4-digit PIN
  POST /api/auth/change-pin   - Change PIN (requires current PIN)
  GET  /api/auth/profile      - Get current user profile

================================================================================
  CRON ENGINE (cron/financialCron.js)
================================================================================

Standalone midnight cron module (runs daily at 00:00)

Two jobs:
  1. autoGenerateInvoices()
     - Reads billingDay from settings table
     - Only runs when today === billingDay (prevents duplicate generation)
     - Generates invoices for all active contracts
     - Uses INV-YYYY-MM-SSS format with per-month sequence
     - Arabic console logging

  2. escalateOverdueInvoices()
     - Finds Unpaid/Partial invoices past due_date
     - Changes status to Overdue
     - Logs summary of escalated invoices

================================================================================
  KEY DESIGN DECISIONS
================================================================================

1. better-sqlite3 (synchronous) - no callback complexity; fits single-tenant
2. Separate payments table (not denormalized) - full audit trail, partial
   payment support
3. Instant invoice creation on contract activation - first invoice visible
   immediately via db.transaction()
4. Tenant DELETE is soft delete (is_former=1) - preserves historical refs
5. Property DELETE requires typed Arabic confirmation word - prevents
   accidental deletion by elderly users
6. All updated_at via AFTER UPDATE triggers - no application code needed
7. Overpayments tracked via invoice notes - no schema changes; future
   invoices can check for pending credits
8. Cron as standalone module in cron/ - separation of concerns, testability
9. Billing day check via settings table - prevents duplicate month generation
