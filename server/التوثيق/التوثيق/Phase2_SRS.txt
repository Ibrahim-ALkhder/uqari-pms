1. INTRODUCTION
================================================================================

1.1 PURPOSE
-----------
This Software Requirements Specification (SRS) defines the complete functional
and non-functional requirements for Phase 2 of the Property Management System.
It supersedes all local/offline constraints from Phase 1. The system is now a
cloud-hosted web application accessed via a web browser on any device (tablet,
laptop, or smartphone). All data is stored on the server, all file uploads go
to server storage, and all automated billing runs server-side via scheduled
cron jobs.

The document is intended for the development team (frontend React developer,
backend Node.js developer, QA engineer) and serves as the single source of
truth for implementation.

1.2 SCOPE
----------
Phase 2 delivers a single-landlord web application with the following
capabilities:
- Secure authentication (email + password + optional PIN).
- Full property, unit, and tenant management.
- Automatic monthly invoice generation via server-side cron job.
- Manual payment collection recording.
- Expense tracking with receipt image uploads.
- Maintenance ticket management with photo uploads.
- Dashboard with real-time statistics.
- Multi-tenant-ready data model (tenant profiles are structured to link to
  future user accounts).
- Settings management (currency, billing day, due day, font size).

Phase 2 explicitly does NOT include:
- Tenant self-service portal.
- Online payment gateway.
- SMS/WhatsApp notifications.
- Multi-language toggle.
- Advanced reporting with charts.

1.3 DEFINITIONS & ACRONYMS
---------------------------
- PMS: Property Management System.
- Landlord: The single authenticated user of the system.
- Tenant: A person renting a unit; managed manually by the landlord; no
  account yet.
- Unit: A rentable space (apartment, shop, room, villa, studio) within a
  property.
- Property: A building or complex containing one or more units.
- Contract: A rental agreement linking a tenant to a unit with a start date,
  end date, and rent amount.
- Invoice: A monthly billing record auto-generated by the cron job.
- CRON: A time-based job scheduler on the server.
- SDG: Sudanese Pound.
- SAR: Saudi Riyal.
- RTL: Right-to-Left text direction.
- JWT: JSON Web Token for authentication.

1.4 TECHNOLOGY STACK
---------------------
Frontend:
  - React 18+ with Vite or Create React App.
  - React Router v6 for client-side routing.
  - Axios for HTTP API communication.
  - CSS Modules or Tailwind CSS (with RTL support).
  - React Query (TanStack Query) for server state management and caching.
  - React Hook Form for form handling with validation.

Backend:
  - Node.js 20+ LTS.
  - Express.js 4+ for REST API.
  - better-sqlite3 (synchronous, fast SQLite driver for Node.js).
  - Multer for file upload handling.
  - node-cron or node-schedule for scheduled tasks.
  - bcryptjs for password hashing.
  - jsonwebtoken for JWT generation and verification.
  - express-validator for input validation and sanitization.

Database:
  - SQLite 3 (via better-sqlite3).
  - Single file: pms_database.db stored in the server's data directory.
  - WAL (Write-Ahead Logging) mode enabled for better concurrent performance.

File Storage:
  - Server local filesystem under /uploads/ directory.
  - Subdirectories: /uploads/receipts/, /uploads/maintenance/.
  - File naming convention: {timestamp}_{original_filename}.
  - Database stores only the relative file path as text.

UC-01: User Login (Landlord Authentication)
----------------------------------------------------------------------

Use Case ID: UC-01
Title: تسجيل الدخول (User Login)
Actor: Landlord (Single user in Phase 2; designed for multi-user expansion)
Description: The landlord authenticates using their email and password to
access the PMS. A successful login issues a JWT token stored in the browser
localStorage. An optional 4-digit PIN can be enabled for an additional
layer of security after login.

Pre-conditions:
  - The landlord account must exist in the database (seeded during initial
    setup by the developer or via a one-time registration screen).
  - The server must be running and accessible.
  - The browser must support localStorage for token persistence.

Basic Flow:
  1. The landlord opens the PMS URL in a web browser on any device.
  2. React detects no valid JWT token in localStorage and redirects to the
     login screen.
  3. The login screen renders an Arabic-language form with:
     - Label "البريد الإلكتروني" (Email) with an email input field.
     - Label "كلمة المرور" (Password) with a password input field (type
       password, with a show/hide toggle button).
     - A large button "تسجيل الدخول" (Login) with primary color (#1A2E40).
     - Help text below the form: "بيانات الدخول خاصة بك. لا تشاركها مع
       أحد." (Your login credentials are private. Do not share them with
       anyone.)
  4. The landlord enters their email and password.
  5. The landlord clicks "تسجيل الدخول".
  6. React frontend sends a POST request to /api/auth/login with the JSON
     body { email, password }.
  7. Backend Express.js receives the request.
  8. Backend validates input format:
     - Email must be a valid email format (regex: /^[^\s@]+@[^\s@]+\.[^\s@]+$/).
     - Password must be at least 6 characters long.
     - If validation fails, return HTTP 400 with { success: false, message:
       "البريد الإلكتروني أو كلمة المرور غير صالحة" (Invalid email or
       password) }.
  9. Backend queries the users table for the email using
     db.prepare('SELECT * FROM users WHERE email = ?').get(email).
  10. If no user is found, return HTTP 401 with { success: false, message:
      "البريد الإلكتروني أو كلمة المرور غير صحيحة" (Email or password is
      incorrect) }. Note: Use the same generic message for both not-found
      and wrong-password to prevent user enumeration.
  11. If user is found, compare the provided password against the stored
      bcrypt hash using bcrypt.compare(password, user.password_hash).
  12. If passwords do not match, return HTTP 401 with the same generic
      error message as step 10.
  13. If passwords match, generate a JWT token using
      jwt.sign({ userId: user.id, role: user.role }, process.env.JWT_SECRET,
      { expiresIn: '7d' }).
  14. Backend returns HTTP 200 with:
      {
        success: true,
        token: "eyJhbGciOiJIUzI1NiIs...",
        user: {
          id: 1,
          name: "أحمد محمد",
          email: "ahmed@example.com",
          role: "landlord",
          hasPin: false
        }
      }
  15. React frontend receives the response.
  16. Frontend stores the token in localStorage under key 'pms_token'.
  17. Frontend stores the user object in localStorage under key 'pms_user'.
  18. Frontend configures Axios default header:
      axios.defaults.headers.common['Authorization'] = 'Bearer ' + token.
  19. Frontend checks the user.hasPin field:
      - If false: redirect to the dashboard (/dashboard).
      - If true: redirect to the PIN entry screen (/pin-entry).
  20. On the PIN entry screen, the landlord enters a 4-digit PIN on a
      numeric keypad (large buttons, touch-friendly).
  21. Frontend sends POST /api/auth/verify-pin with { pin: "1234" }.
  22. Backend verifies the PIN against the stored hash for this user.
  23. If PIN is correct, frontend redirects to dashboard.
  24. If PIN is incorrect after 3 attempts, the account is locked for 5
      minutes (stored in memory or a login_attempts table). The screen
      shows "تم قفل الدخول مؤقتاً. حاول بعد 5 دقائق." (Login locked
      temporarily. Try again in 5 minutes.)

Alternative Flows:
  - AF-01: Network Error – If the POST request fails due to network issues,
    React displays an error banner at the top of the login form:
    "تعذر الاتصال بالخادم. تحقق من اتصالك بالإنترنت." (Could not connect
    to the server. Check your internet connection.)
  - AF-02: Token Expired – If the API returns HTTP 401 on any subsequent
    request (token expired), React intercepts this globally via Axios
    interceptor. It clears localStorage and redirects to the login screen
    with a message: "انتهت صلاحية الجلسة. يرجى تسجيل الدخول مرة أخرى."
    (Session expired. Please log in again.)
  - AF-03: Invalid PIN 3 Times – Count failed PIN attempts in
    login_attempts table (user_id, attempt_count, last_attempt_at).
    On the 3rd failure, set locked_until = NOW() + 5 minutes. Return
    HTTP 429 too many requests. Frontend displays the lockout message.
  - AF-04: Browser Does Not Support localStorage – Show an error screen:
    "هذا المتصفح لا يدعم التطبيق. يرجى استخدام متصفح حديث مثل Google
    Chrome أو Firefox." (This browser does not support the application.
    Please use a modern browser such as Google Chrome or Firefox.)

Post-conditions:
  - A valid JWT token is stored in the browser's localStorage.
  - Axios global authorization header is set.
  - The landlord is redirected to the Dashboard screen.
  - A session log entry is written to the audit_log table (user_id, action:
    'LOGIN', ip_address, user_agent, created_at).

Validation Rules:
  - Email: Required; must be a valid email format; maximum 255 characters.
  - Password: Required; minimum 6 characters; maximum 128 characters.
  - PIN: Exactly 4 digits (0-9); stored as bcrypt hash.

Arabic UI Labels & Messages (must be rendered in the React component):
  - Screen title: "تسجيل الدخول إلى النظام" (Login to the System)
  - Email field label: "البريد الإلكتروني"
  - Email placeholder: "example@email.com"
  - Password field label: "كلمة المرور"
  - Password placeholder: "أدخل كلمة المرور"
  - Login button: "تسجيل الدخول"
  - Error (invalid credentials): "البريد الإلكتروني أو كلمة المرور غير صحيحة"
  - Error (network): "تعذر الاتصال بالخادم. تحقق من اتصالك بالإنترنت."
  - Error (session expired): "انتهت صلاحية الجلسة. يرجى تسجيل الدخول مرة أخرى."
  - PIN screen title: "أدخل الرقم السري"
  - PIN error: "الرقم السري غير صحيح"
  - PIN lockout: "تم قفل الدخول مؤقتاً. حاول بعد 5 دقائق."
  - Show password toggle: "إظهار" / "إخفاء"

UC-02: Add Property
----------------------------------------------------------------------

Use Case ID: UC-02
Title: إضافة عقار جديد (Add New Property)
Actor: Landlord (authenticated)
Description: The landlord adds a new property (building/complex) to the
system. Each property acts as a container for rental units.

Pre-conditions:
  - The landlord must be authenticated (valid JWT token in Authorization
    header).
  - The landlord must be on the Properties list screen.

Basic Flow:
  1. The landlord navigates to the Properties screen by tapping
     "العقارات" (Properties) on the main navigation menu.
  2. The Properties list screen displays all existing properties as large
     cards. Each card shows the property name, city, number of units, and
     an edit/delete button.
  3. At the bottom of the screen, a large button reads
     "إضافة عقار جديد" (Add New Property).
  4. The landlord taps the "إضافة عقار جديد" button.
  5. React renders the Add Property form screen with the following fields
     (all labels and placeholder text in Arabic):
     a. "اسم العقار" (Property Name) – text input; required.
        Placeholder: "مثال: عمارة الملك فهد"
     b. "المدينة / الحي" (City / Neighborhood) – text input; required.
        Placeholder: "مثال: الرياض - حي النزهة"
     c. "عدد الوحدات" (Number of Units) – number input (type="number",
        min="1", max="100"); required. Placeholder: "مثال: 6"
     d. "ملاحظات" (Notes) – textarea; optional.
        Placeholder: "أي ملاحظات إضافية (اختياري)"
     e. A help text banner at the top of the form: "سيتم إنشاء الوحدات
        تلقائياً بعد إضافة العقار. يمكنك تعديلها لاحقاً." (Units will be
        created automatically after adding the property. You can edit
        them later.)
     f. Two buttons at the bottom of the form:
        - "حفظ" (Save) – primary button, color #1A2E40.
        - "إلغاء" (Cancel) – secondary button, color gray.
  6. The landlord fills in the form fields.
  7. The landlord taps "حفظ" (Save).
  8. React frontend performs client-side validation:
     - Property name: required, minimum 2 characters, maximum 100 characters.
     - City: required, minimum 2 characters, maximum 100 characters.
     - Number of units: required, must be an integer between 1 and 100.
     - Notes: optional, maximum 500 characters.
     - If validation fails, React highlights the invalid field in red and
       shows the corresponding error message below the field.
  9. If client-side validation passes, React sends a POST request to
     /api/properties with the JSON body:
     {
       name: "عمارة الملك فهد",
       city: "الرياض - حي النزهة",
       unitCount: 6,
       notes: ""
     }
     The Authorization header includes 'Bearer ' + token.
  10. Backend receives the POST request.
  11. Backend middleware checks the JWT token validity and extracts userId
      from the token payload. If invalid/expired, return HTTP 401.
  12. Backend validates the input data on the server side using
      express-validator:
      - name: notEmpty, isLength({ min: 2, max: 100 }), trim, escape.
      - city: notEmpty, isLength({ min: 2, max: 100 }), trim, escape.
      - unitCount: isInt({ min: 1, max: 100 }), toInt.
      - notes: optional, isLength({ max: 500 }), trim, escape.
      - If validation fails, return HTTP 422 with:
        { success: false, errors: [{ field: "name", message: "اسم العقار
          مطلوب ويجب ألا يقل عن حرفين" }] }
  13. If validation passes, backend begins a database transaction:
      const transaction = db.transaction(() => { ... });
  14. Insert the property into the properties table:
      const propertyResult = db.prepare(
        'INSERT INTO properties (user_id, name, city, notes, created_at,
         updated_at) VALUES (?, ?, ?, ?, datetime("now"), datetime("now"))'
      ).run(userId, name, city, notes);
      const propertyId = propertyResult.lastInsertRowid;
  15. Generate unitCount units for this property. Loop from 1 to unitCount
      and insert each unit:
      for (let i = 1; i <= unitCount; i++) {
        db.prepare(
          'INSERT INTO units (property_id, unit_number, type, monthly_rent,
           status, created_at, updated_at) VALUES (?, ?, ?, ?, ?, datetime
           ("now"), datetime("now"))'
        ).run(propertyId, `وحدة ${i}`, 'Apartment', 0, 'Vacant');
      }
      Note: Default unit names are Arabic: "وحدة 1", "وحدة 2", etc.
      The landlord can edit these later.
  16. Transaction commits.
  17. Backend returns HTTP 201 with:
      {
        success: true,
        message: "تم إضافة العقار بنجاح" (Property added successfully),
        data: {
          id: propertyId,
          name: "عمارة الملك فهد",
          city: "الرياض - حي النزهة",
          unitCount: 6,
          units: [ { id: 1, unitNumber: "وحدة 1" }, ... ]
        }
      }
  18. React frontend receives the response.
  19. Frontend shows a green success toast notification at the top of the
      screen: "✓ تم إضافة العقار بنجاح" for 3 seconds.
  20. Frontend navigates back to the Properties list screen.
  21. The Properties list screen refetches the property list from the API
      (GET /api/properties) and displays the updated list with the new
      property included.

Alternative Flows:
  - AF-01: Duplicate Property Name – Backend checks if a property with the
    same name already exists for this user. If yes, return HTTP 409 with
    { success: false, message: "يوجد عقار بنفس الاسم بالفعل. اختر اسماً
    آخر." (A property with the same name already exists. Choose another
    name.) } The frontend displays this error below the name field.
  - AF-02: Server Error – If an unexpected error occurs during the database
    transaction, the transaction rolls back automatically. Return HTTP 500
    with { success: false, message: "حدث خطأ في الخادم. الرجاء المحاولة
    لاحقاً." (A server error occurred. Please try again later.) }
  - AF-03: Network Error – Same handling as UC-01 AF-01. A banner displays
    the connection error message.
  - AF-04: User Cancels – If the landlord taps "إلغاء" (Cancel), the form
    is discarded and the user is navigated back to the Properties list
    screen. No data is sent to the server.

Post-conditions:
  - A new property record exists in the properties table with the specified
    details.
  - N new unit records exist in the units table (where N = unitCount), all
    with status 'Vacant' and monthly_rent set to 0.
  - The property appears in the Properties list.
  - An audit log entry is created: action: 'ADD_PROPERTY', target_id:
    propertyId.

Validation Rules:
  - Property name: Required; 2-100 characters; no leading/trailing whitespace.
  - City: Required; 2-100 characters.
  - Unit count: Required; integer; min 1, max 100.
  - Notes: Optional; max 500 characters; HTML tags stripped.

Arabic UI Labels & Messages:
  - Screen title: "إضافة عقار جديد"
  - Field label: "اسم العقار"
  - Field placeholder: "مثال: عمارة الملك فهد"
  - Field label: "المدينة / الحي"
  - Field placeholder: "مثال: الرياض - حي النزهة"
  - Field label: "عدد الوحدات"
  - Field placeholder: "مثال: 6"
  - Field label: "ملاحظات"
  - Field placeholder: "أي ملاحظات إضافية (اختياري)"
  - Help banner: "سيتم إنشاء الوحدات تلقائياً بعد إضافة العقار. يمكنك
    تعديلها لاحقاً."
  - Save button: "حفظ"
  - Cancel button: "إلغاء"
  - Success toast: "✓ تم إضافة العقار بنجاح"
  - Error (duplicate name): "يوجد عقار بنفس الاسم بالفعل. اختر اسماً آخر."
  - Error (server): "حدث خطأ في الخادم. الرجاء المحاولة لاحقاً."
  - Error (validation name): "اسم العقار مطلوب ويجب ألا يقل عن حرفين"
  - Error (validation city): "المدينة مطلوبة"
  - Error (validation units): "عدد الوحدات يجب أن يكون بين 1 و 100"

UC-03: Add Unit
----------------------------------------------------------------------

Use Case ID: UC-03
Title: إضافة وحدة جديدة (Add New Unit)
Actor: Landlord (authenticated)
Description: The landlord adds a new rental unit to an existing property.
Units represent individual rentable spaces such as apartments, shops,
rooms, villas, or studios.

Pre-conditions:
  - The landlord must be authenticated.
  - At least one property must exist in the system.
  - The landlord must be viewing the details screen of a specific property.

Basic Flow:
  1. The landlord navigates to a property's detail screen by tapping on
     a property card from the Properties list.
  2. The property detail screen displays the property info at the top and
     a list of its units below. Each unit card shows:
     - Unit number (e.g., "وحدة 1").
     - Unit type (e.g., "شقة").
     - Monthly rent in large bold text (e.g., "٢٬٠٠٠ ر.س" or "٠ ر.س").
     - Status badge: "شاغرة" (Vacant) in green, "مؤجرة" (Occupied) in blue,
       or "تحت الصيانة" (Under Maintenance) in orange.
  3. At the bottom of the unit list, a large button reads
     "إضافة وحدة جديدة" (Add New Unit).
  4. The landlord taps "إضافة وحدة جديدة".
  5. React renders the Add Unit form as a modal overlay or a new screen
     with the following fields:
     a. "رقم الوحدة" (Unit Number) – text input; required.
        Placeholder: "مثال: 101 أو شقة 3"
     b. "نوع الوحدة" (Unit Type) – dropdown; required. Options:
        - "شقة" (Apartment)
        - "محل" (Shop)
        - "غرفة" (Room)
        - "فيلا" (Villa)
        - "استوديو" (Studio)
     c. "رقم الطابق" (Floor Number) – number input; optional.
        Placeholder: "مثال: 2"
     d. "الإيجار الشهري" (Monthly Rent) – number input; required.
        Placeholder: "مثال: 2000"
        The input should show a currency suffix based on settings
        ("ر.س" for SAR, "ج.س" for SDG).
     e. "الحالة" (Status) – dropdown; default value "شاغرة" (Vacant).
        Options: "شاغرة" (Vacant), "مؤجرة" (Occupied),
        "تحت الصيانة" (Under Maintenance).
     f. Two buttons: "حفظ" (Save) and "إلغاء" (Cancel).
  6. The landlord fills in the form fields.
  7. The landlord taps "حفظ".
  8. React performs client-side validation:
     - Unit number: required; 1-50 characters.
     - Unit type: required; must be one of the predefined enum values.
     - Floor number: optional; must be an integer if provided.
     - Monthly rent: required; must be a non-negative number; max 999999999.
     - Status: required; must be one of 'Vacant', 'Occupied',
       'Under Maintenance'.
  9. If validation passes, React sends POST /api/properties/:propertyId/units
     with the JSON body:
     {
       unitNumber: "101",
       type: "Apartment",
       floor: 2,
       monthlyRent: 2000,
       status: "Vacant"
     }
  10. Backend validates the request:
      - propertyId must be a valid integer and must exist in the properties
        table for this user.
      - Unit number must be unique within this property (checked via
        db query: SELECT id FROM units WHERE property_id = ? AND
        unit_number = ?).
      - All field validations as per client side.
  11. If validation fails, return appropriate HTTP 422 or 409.
  12. If validation passes, insert into units table:
      db.prepare(
        'INSERT INTO units (property_id, unit_number, type, floor,
         monthly_rent, status, created_at, updated_at) VALUES (?, ?, ?, ?,
         ?, ?, datetime("now"), datetime("now"))'
      ).run(propertyId, unitNumber, type, floor || null, monthlyRent, status);
  13. Get the lastInsertRowid as the new unit ID.
  14. Backend returns HTTP 201 with the created unit data.
  15. React shows success toast "✓ تم إضافة الوحدة بنجاح" and refreshes
      the unit list.
  16. The new unit appears in the unit list for the property.

Alternative Flows:
  - AF-01: Duplicate Unit Number in Same Property – Backend returns
    HTTP 409 with message: "رقم الوحدة موجود مسبقاً في هذا العقار. اختر
    رقماً آخر." (Unit number already exists in this property. Choose
    another number.) Frontend displays the error below the unit number
    field.
  - AF-02: Property Does Not Exist – If propertyId does not exist or
    belongs to another user, return HTTP 404 with message:
    "العقار غير موجود." (Property not found.)
  - AF-03: Server Error – Return HTTP 500, transaction rollback if
    applicable, display generic server error.
  - AF-04: User Cancels – Return to property detail screen without changes.

Post-conditions:
  - A new unit record exists in the units table linked to the specified
    property.
  - The property's unit count (calculated dynamically via COUNT query)
    increments by 1.
  - The unit appears in the property detail screen.
  - Audit log entry: action 'ADD_UNIT', target_id: unitId.

Validation Rules:
  - Unit number: Required; 1-50 characters; unique within the same property.
  - Unit type: Required; enum: 'Apartment', 'Shop', 'Room', 'Villa',
    'Studio'.
  - Floor: Optional; integer; 0 to 200.
  - Monthly rent: Required; number; min 0; max 999999999.
  - Status: Required; enum: 'Vacant', 'Occupied', 'Under Maintenance'.

Arabic UI Labels & Messages:
  - Screen title: "إضافة وحدة جديدة"
  - Field label: "رقم الوحدة"

UC-05: Activate Contract & Generate Invoices
----------------------------------------------------------------------

Use Case ID: UC-05
Title: تفعيل العقد وتوليد الفواتير (Activate Contract & Generate Invoices)
Actor: Landlord (authenticated) + Cron Job (automated)
Description: A contract is automatically created when a tenant is registered
to a unit. The contract defines the rental terms. The system's cron job runs
daily at midnight and generates invoices for all active contracts. The
landlord can also view and manage contracts manually.

Pre-conditions (Contract Creation):
  - A tenant has been registered to a unit (UC-04).
  - The contract is created automatically as part of UC-04, step 12.

Pre-conditions (Invoice Generation):
  - The server's cron job must be configured to run at 00:00 server time
    daily. (node-cron: '0 0 * * *').
  - At least one active contract must exist in the database.

Basic Flow (Contract Creation – Automatic via UC-04):
  1. A new contract is inserted into the contracts table with:
     - tenant_id: the newly created tenant's ID.
     - unit_id: the assigned unit's ID.
     - start_date: CURRENT_DATE (the date tenant was registered).
     - end_date: NULL (open-ended; landlord can set an end date later).
     - monthly_rent: the unit's monthly_rent value at time of assignment.
     - status: 'Active'.
     - created_at, updated_at: current timestamps.
  2. A confirmation is returned in the UC-04 API response.
  3. The landlord can also view all contracts via a Contracts screen:
    "العقود" (Contracts) in the navigation menu.
  4. The Contracts list screen shows: tenant name, unit info, start date,
     monthly rent, and status ("نشط" Active, "منتهي" Expired).
  5. The landlord can tap a contract to view details.
  6. The landlord can terminate a contract early by tapping
     "إنهاء العقد" (Terminate Contract). A confirmation dialog appears:
     "سيتم إنهاء العقد وتحويل الوحدة إلى شاغرة. هل أنت متأكد؟"
     (The contract will be terminated and the unit will become vacant.
     Are you sure?)
  7. If confirmed, backend:
     - Updates contract status to 'Expired' and sets end_date = today.
     - Updates unit status to 'Vacant'.
     - Returns success.

Basic Flow (Invoice Generation – Automated Cron Job):
  1. The server starts and the cron job is registered in the Express app
     initialization:
     const cron = require('node-cron');
     cron.schedule('0 0 * * *', async () => {
       console.log('[CRON] Running daily invoice generation:', new Date());
       await generateMonthlyInvoices();
     });
  2. The generateMonthlyInvoices() function executes:
     a. Determine the current month and year (e.g., June 2026).
     b. Query all active contracts:
        SELECT c.*, u.monthly_rent, u.unit_number, p.name as property_name,
        t.full_name as tenant_name
        FROM contracts c
        JOIN units u ON c.unit_id = u.id
        JOIN properties p ON u.property_id = p.id
        JOIN tenants t ON c.tenant_id = t.id
        WHERE c.status = 'Active'
        AND (c.end_date IS NULL OR c.end_date >= date('now'))
     c. For each active contract:
        i. Check if an invoice already exists for this contract for the
           current month:
           SELECT id FROM invoices WHERE contract_id = ? AND
           billing_month = ? AND billing_year = ?
        ii. If an invoice already exists, skip (idempotent – prevents
            duplicate invoices if the cron job runs multiple times).
        iii. If no invoice exists, generate one:
             INSERT INTO invoices (contract_id, unit_id, tenant_id,
             invoice_number, billing_month, billing_year, amount,
             due_date, status, created_at, updated_at)
             VALUES (?, ?, ?, ?, ?, ?, ?, ?, 'Unpaid', datetime("now"),
             datetime("now"))
             - invoice_number: auto-generated as "INV-{YEAR}-{MONTH}-{SEQ}".
               The sequence is a 3-digit zero-padded number that resets
               each month. Calculated as: SELECT COUNT(*) + 1 FROM invoices
               WHERE billing_month = ? AND billing_year = ?.
             - billing_month: month number (1-12).
             - billing_year: 4-digit year.
             - amount: from contract.monthly_rent.
             - due_date: calculated as: {billing_year}-{billing_month}-
               {settings.due_day}. If the due day is beyond the month's
               last day, use the last day of the month. For example, if
               due_day = 31 and the month has 30 days, due_date = 30th.
     d. Log the number of invoices generated:
        console.log(`[CRON] Generated ${count} invoices for
        ${month}/${year}`);
  3. Invoices are now available in the database.
  4. When the landlord opens the Invoices screen, React fetches
     GET /api/invoices?month=6&year=2026.
  5. The backend returns all invoices for the specified month, including
     tenant name, unit number, amount, due date, and status.
  6. The frontend renders the invoice list grouped by status
     (Unpaid first, then Overdue, then Paid).

Basic Flow (Manual Contract View & Termination):
  1. The landlord navigates to "العقود" (Contracts) screen.
  2. Backend returns GET /api/contracts with all contracts for this user,
     joining tenant and unit data.
  3. Frontend renders a list of contract cards. Each card shows:
     - Tenant name (large bold text).
     - Unit: "عمارة الملك فهد - وحدة 101".
     - Monthly rent: "٢٬٠٠٠ ر.س".
     - Status badge: "نشط" (green) or "منتهي" (gray).
     - Start date: "١ يونيو ٢٠٢٦".
  4. Tapping a card opens the contract detail screen.
  5. The contract detail screen shows all contract fields and two action
     buttons:
     - "إنهاء العقد" (Terminate Contract) – red button.
     - "عرض الفواتير" (View Invoices) – navigates to invoices filtered
       by this contract.
  6. Tapping "إنهاء العقد" triggers the termination flow described above.

Alternative Flows:
  - AF-01: Cron Job Fails – If the cron job throws an unhandled exception,
    it logs an error: "[CRON ERROR] Invoice generation failed: {error}".
    The cron job does NOT retry automatically; it waits for the next
    scheduled run. A manual "Generate Invoices Now" button exists in the
    Settings or Invoice screen (Admin only) that the developer or landlord
    can trigger to run the generation function on demand.
  - AF-02: No Active Contracts – If the SELECT query returns zero rows,
    the cron job logs "[CRON] No active contracts found. No invoices
    generated." and exits cleanly.
  - AF-03: Contract Termination with Unpaid Invoices – If the landlord
    terminates a contract that has unpaid invoices, a warning dialog
    appears: "يوجد فواتير غير مدفوعة لهذا العقد. هل تريد إنهاء العقد
    مع بقاء الفواتير غير مدفوعة؟" (There are unpaid invoices for this
    contract. Do you want to terminate the contract with the invoices
    remaining unpaid?) Options: "نعم، إنهاء العقد" and "إلغاء".
    If the landlord proceeds, the contract is terminated but invoices
    remain in the system with their unpaid status.
  - AF-04: Duplicate Invoice Prevention – The check in step 2.c.i ensures
    idempotency. Even if the server restarts mid-cron and the job runs
    twice, no duplicate invoices are created.
  - AF-05: Settings Due Day Adjustment – If the landlord changes the due
    day in settings (e.g., from 5 to 10), the change only affects
    invoices generated from the next billing cycle onward. Already-
    generated invoices retain their original due dates.

Post-conditions:
  - A new invoice record exists in the invoices table for each active
    contract for the current month.
  - The invoice list screen shows the new invoices with status 'Unpaid'.
  - The Dashboard's "Total Expected Monthly Rent" and "Invoices Due Soon"
    sections are updated (calculated from the database on each page load).
  - Audit log entry: 'BULK_INVOICE_GENERATION' with count of invoices
    generated.
  - (If terminated) Contract status = 'Expired', unit status = 'Vacant'.

Validation Rules:
  - Contract start_date cannot be in the future (must be today or past).
  - Contract end_date must be after start_date if provided.
  - Invoice amount must equal the contract's monthly_rent (from the unit).
  - Invoice billing_month must be 1-12; billing_year must be 2020-2099.

Arabic UI Labels & Messages:
  - Screen title: "العقود" (Contracts)
  - Card labels: "المستأجر:" (Tenant:), "الوحدة:" (Unit:), "الإيجار:"
    (Rent:), "منذ:" (Since:)
  - Status active: "نشط"
  - Status expired: "منتهي"
  - Button: "إنهاء العقد"
  - Button: "عرض الفواتير"
  - Confirmation terminate: "سيتم إنهاء العقد وتحويل الوحدة إلى شاغرة.
    هل أنت متأكد؟"
  - Warning unpaid: "يوجد فواتير غير مدفوعة لهذا العقد. هل تريد إنهاء
    العقد مع بقاء الفواتير غير مدفوعة؟"
  - Success terminate: "✓ تم إنهاء العقد بنجاح"
  - Invoice screen title: "الفواتير"
  - Invoice statuses: "غير مدفوعة", "مدفوعة", "مدفوعة جزئياً", "متأخرة"
  - Button: "توليد الفواتير الآن" (Generate Invoices Now – manual trigger)
  - Invoice number format displayed: "رقم الفاتورة: INV-2026-06-001"

UC-06: Collect Invoice Payment
----------------------------------------------------------------------

Use Case ID: UC-06
Title: تحصيل دفعة فاتورة (Collect Invoice Payment)
Actor: Landlord (authenticated)
Description: The landlord records a payment received from a tenant against
a specific invoice. Supports full payment, partial payment, and multiple
payment methods.

Pre-conditions:
  - At least one invoice with status 'Unpaid' or 'Overdue' must exist.
  - The landlord must be on the Invoice List screen or Invoice Detail screen.

Basic Flow:
  1. The landlord navigates to "الفواتير" (Invoices) from the main
     navigation.
  2. The Invoice List screen shows all invoices for the current month,
     organized in sections:
     - "متأخرة" (Overdue) – Red section header.
     - "غير مدفوعة" (Unpaid) – Yellow section header.
     - "مدفوعة" (Paid) – Green section header.
  3. Each invoice card shows:
     - Invoice number (e.g., "INV-2026-06-001").
     - Tenant name (large text).
     - Unit info.
     - Amount (bold).
     - Due date.
     - Status badge (colored).
  4. The landlord taps on an unpaid or overdue invoice.
  5. React navigates to the Invoice Detail screen.
  6. The Invoice Detail screen shows full invoice information and a large
     button: "تسجيل الدفع" (Record Payment).
  7. The landlord taps "تسجيل الدفع".
  8. A payment modal/form appears with the following fields:
     a. "المبلغ المدفوع" (Amount Paid) – number input; pre-filled with
        the full invoice amount. The landlord can reduce it for partial
        payments. Required.
     b. "تاريخ الدفع" (Payment Date) – date input; pre-filled with today's
        date. Required.
     c. "طريقة الدفع" (Payment Method) – dropdown; required. Options:
        - "نقداً" (Cash)
        - "تحويل بنكي" (Bank Transfer)
        - "شيك" (Cheque)
     d. "ملاحظات" (Notes) – text; optional. Placeholder: "ملاحظات إضافية
        (اختياري)"
     e. Two buttons: "تأكيد الدفع" (Confirm Payment) and "إلغاء" (Cancel).
  9. The landlord enters the amount (e.g., full or partial) and other
     details, then taps "تأكيد الدفع".
  10. React validates:
      - Amount must be greater than 0.
      - Amount must not exceed the invoice amount.
      - Payment date must not be in the future.
  11. React sends POST /api/invoices/:invoiceId/payments with:
      {
        amount: 2000,
        paymentDate: "2026-06-05",
        paymentMethod: "Cash",
        notes: ""
      }
  12. Backend validates:
      - Invoice exists and belongs to this user.
      - Invoice is not already fully paid.
      - Amount > 0 and <= the remaining balance (invoice amount minus
        sum of all previous payments for this invoice).
      - Payment date is valid (not future).
  13. Backend begins a transaction:
      - Insert into payments table:
        INSERT INTO payments (invoice_id, amount, payment_date,
        payment_method, notes, created_at) VALUES (?, ?, ?, ?, ?,
        datetime("now"))
      - Calculate the total paid so far for this invoice:
        SELECT COALESCE(SUM(amount), 0) FROM payments WHERE
        invoice_id = ?
      - Determine the new invoice status:
        * If totalPaid >= invoice.amount → 'Paid'
        * If totalPaid > 0 AND totalPaid < invoice.amount → 'Partial'
        * If totalPaid = 0 → no change (should not happen here)
      - Update the invoice status:
        UPDATE invoices SET status = ?, updated_at = datetime("now")
        WHERE id = ?
  14. Commit transaction.
  15. Backend returns HTTP 200 with updated invoice and payment data:
      {
        success: true,
        message: "✓ تم تسجيل الدفع بنجاح" (Payment recorded successfully),
        data: {
          invoice: { id: 5, status: "Paid", ... },
          payment: { id: 1, amount: 2000, ... }
        }
      }
  16. React updates the invoice detail screen. The status badge changes to
      "مدفوعة" (Paid) or "مدفوعة جزئياً" (Partial) with the remaining
      balance shown.
  17. A green success toast is shown for 3 seconds.
  18. The Dashboard's financial summary updates on next fetch.

Alternative Flows:
  - AF-01: Partial Payment – The landlord enters an amount less than the
    invoice total. The invoice status becomes 'Partial'. The remaining
    balance is displayed on the invoice card: "المتبقي: ٥٠٠ ر.س"
    (Remaining: 500 SAR). The landlord can record additional payments
    against the same invoice later.
  - AF-02: Overpayment – If the entered amount exceeds the remaining
    balance, the backend returns HTTP 422 with:
    "المبلغ المدفوع أكبر من المبلغ المتبقي. المتبقي: {amount}."
    (Amount exceeds remaining balance. Remaining: {amount}.)
  - AF-03: Invoice Already Fully Paid – If the invoice status is already
    'Paid', the "تسجيل الدفع" button is disabled/hidden. A message reads:
    "هذه الفاتورة مدفوعة بالكامل." (This invoice is fully paid.)
  - AF-04: Payment on Overdue Invoice – The same flow applies. The invoice
    status transitions from 'Overdue' to 'Paid' or 'Partial'. The overdue
    counter on the Dashboard updates accordingly.
  - AF-05: Multiple Payments on Same Invoice – The system supports
    multiple partial payments. Each payment is recorded separately. The
    total paid is the sum of all payments. The remaining balance is
    invoice.amount - SUM(payments.amount).
  - AF-06: Server Error – Rollback transaction, return HTTP 500.

Post-conditions:
  - A new payment record exists in the payments table.
  - The invoice status is updated ('Paid', 'Partial', or remains 'Overdue'
    if partial but past due date – note: if partial AND past due, status
    stays 'Overdue' or a combined status 'Partial_Overdue' – decision:
    use 'Partial' and let the dashboard logic detect that due date is past
    + status is Partial to flag it).
  - The payment appears in the invoice detail payment history list.
  - The Dashboard's monthly collection total increases.
  - Audit log entry: 'ADD_PAYMENT' with invoice ID and amount.

Validation Rules:

UC-09: Dashboard Overview
----------------------------------------------------------------------

Use Case ID: UC-09
Title: لوحة التحكم (Dashboard Overview)
Actor: Landlord (authenticated)
Description: The main landing screen after login. Shows a real-time summary
of key business metrics in a senior-friendly format: large numbers, clear
labels, and high-contrast colors.

Pre-conditions:
  - The landlord must be authenticated.
  - JWT token must be valid.

Basic Flow:
  1. On successful login (and PIN verification if enabled), the React
     frontend navigates to /dashboard.
  2. The React frontend sends a GET request to /api/dashboard/summary.
  3. Backend executes the following queries and combines the results:
     a. Total properties:
        SELECT COUNT(*) as total FROM properties WHERE user_id = ?
     b. Total units:
        SELECT COUNT(*) as total FROM units u JOIN properties p ON
        u.property_id = p.id WHERE p.user_id = ?
     c. Occupied units:
        SELECT COUNT(*) as total FROM units u JOIN properties p ON
        u.property_id = p.id WHERE p.user_id = ? AND u.status = 'Occupied'
     d. Vacant units:
        SELECT COUNT(*) as total FROM units u JOIN properties p ON
        u.property_id = p.id WHERE p.user_id = ? AND u.status = 'Vacant'
     e. Total expected monthly rent (sum of monthly_rent for all occupied
        units):
        SELECT COALESCE(SUM(u.monthly_rent), 0) as total FROM units u
        JOIN properties p ON u.property_id = p.id WHERE p.user_id = ?
        AND u.status = 'Occupied'
     f. Total overdue invoices amount:
        SELECT COALESCE(SUM(i.amount - COALESCE(p.total_paid, 0)), 0) as
        total FROM invoices i JOIN contracts c ON i.contract_id = c.id
        JOIN units u ON c.unit_id = u.id JOIN properties p ON
        u.property_id = p.id LEFT JOIN (SELECT invoice_id, SUM(amount) as
        total_paid FROM payments GROUP BY invoice_id) p ON i.id =
        p.invoice_id WHERE p.user_id = ? AND i.status IN ('Unpaid',
        'Overdue', 'Partial') AND i.due_date < date('now')
        Note: This is simplified. The actual query may use a helper view.
     g. Overdue invoices count:
        SELECT COUNT(*) as count FROM invoices i JOIN contracts c ON
        i.contract_id = c.id JOIN units u ON c.unit_id = u.id JOIN
        properties p ON u.property_id = p.id WHERE p.user_id = ? AND
        i.status IN ('Unpaid', 'Partial') AND i.due_date < date('now')
     h. Invoices due within 7 days (including today):
        SELECT i.id, i.invoice_number, i.amount, i.due_date, t.full_name
        as tenant_name, u.unit_number, p.name as property_name
        FROM invoices i JOIN contracts c ON i.contract_id = c.id
        JOIN units u ON c.unit_id = u.id JOIN properties p ON
        u.property_id = p.id JOIN tenants t ON c.tenant_id = t.id
        WHERE p.user_id = ? AND i.status IN ('Unpaid', 'Partial')
        AND i.due_date BETWEEN date('now') AND date('now', '+7 days')
        ORDER BY i.due_date ASC
        LIMIT 10
     i. Current month total collected:
        SELECT COALESCE(SUM(py.amount), 0) as total FROM payments py
        JOIN invoices i ON py.invoice_id = i.id JOIN contracts c ON
        i.contract_id = c.id JOIN units u ON c.unit_id = u.id JOIN
        properties p ON u.property_id = p.id WHERE p.user_id = ? AND
        strftime('%Y-%m', py.payment_date) = strftime('%Y-%m', 'now')
     j. Current month total expenses:
        SELECT COALESCE(SUM(amount), 0) as total FROM expenses WHERE
        user_id = ? AND strftime('%Y-%m', date) = strftime('%Y-%m', 'now')
  4. Backend returns a combined JSON response with all the above data.
  5. React renders the Dashboard with the following layout (top to bottom):
     a. Header: "مرحباً بك، {user.name}" (Welcome, {user.name}) in 28px

UC-10: Backup & Restore Data
----------------------------------------------------------------------

Use Case ID: UC-10
Title: نسخ احتياطي واستعادة (Backup & Restore Data)
Actor: Landlord (authenticated)
Description: The landlord can export all data to a JSON file for safekeeping
and restore it later. This is a cloud-hosted system, so the backup is
downloaded via the browser as a .json file.

Pre-conditions:
  - The landlord must be authenticated.

Basic Flow – Backup:
  1. The landlord navigates to "الإعدادات" (Settings) via the navigation
     menu.
  2. The Settings screen shows a section titled "النسخ الاحتياطي" (Backup).
  3. A large button reads "نسخ احتياطي للبيانات" (Backup Data).
  4. The landlord taps the button.
  5. React sends GET /api/backup/export.
  6. Backend verifies the JWT token.
  7. Backend queries all tables for this user_id:
     - SELECT * FROM properties WHERE user_id = ?
     - SELECT * FROM units WHERE property_id IN (SELECT id FROM properties
       WHERE user_id = ?)
     - SELECT * FROM tenants WHERE user_id = ?
     - SELECT * FROM contracts WHERE tenant_id IN (SELECT id FROM tenants
       WHERE user_id = ?)
     - SELECT * FROM invoices WHERE contract_id IN (SELECT id FROM contracts
       WHERE tenant_id IN (SELECT id FROM tenants WHERE user_id = ?))
     - SELECT * FROM payments WHERE invoice_id IN (SELECT id FROM invoices
       WHERE contract_id IN (SELECT id FROM contracts WHERE tenant_id IN
       (SELECT id FROM tenants WHERE user_id = ?)))
     - SELECT * FROM expenses WHERE user_id = ?
     - SELECT * FROM maintenance_tickets WHERE user_id = ?
     - SELECT * FROM settings WHERE user_id = ?
  8. Backend packages all data into a single JSON object:
     {
       version: "2.0",
       exportedAt: "2026-06-06T12:00:00.000Z",
       user: { name, email },
       data: { properties, units, tenants, contracts, invoices, payments,
               expenses, maintenanceTickets, settings }
     }
  9. Backend sets response headers:
     Content-Type: application/json
     Content-Disposition: attachment; filename="pms_backup_2026-06-06.json"
  10. Backend sends the JSON file as the response body.
  11. React (or the browser) triggers a file download. The file is saved
      to the device's Downloads folder.
  12. A success toast is shown: "✓ تم إنشاء النسخ الاحتياطي وتحميله."
      (Backup has been created and downloaded.)

Basic Flow – Restore:
  1. On the same Settings screen, below the backup button, a button reads
     "استعادة البيانات من نسخة احتياطية" (Restore Data from Backup).
  2. The landlord taps the button.
  3. A warning dialog appears with red text:
     "⚠ تحذير: سيتم استبدال جميع البيانات الحالية بالبيانات الموجودة في
     النسخة الاحتياطية. لا يمكن التراجع عن هذا الإجراء. هل أنت متأكد؟"
     (WARNING: All current data will be replaced with the data from the
     backup file. This action cannot be undone. Are you sure?)
     Buttons: "نعم، استعادة" (Yes, Restore) and "إلغاء" (Cancel).
  4. The landlord confirms by tapping "نعم، استعادة".
  5. A file picker dialog opens (hidden file input element triggered by
     React). The landlord selects a .json file from their device.
  6. React reads the file and sends it to the server via POST
     /api/backup/restore with FormData containing the file.
  7. Backend validates:
     - File is valid JSON.
     - File has the required structure (version, data, etc.).
     - version field is compatible (major version matches).
     - All user_id references point to the current authenticated user.
     - Data integrity: for example, every unit must reference an existing
       property_id from within the backup.
  8. If validation fails, return HTTP 422 with a specific Arabic error
     message explaining the issue.
  9. If validation passes, backend begins a massive transaction:
     a. Delete all existing data for this user (CASCADE order):
        - DELETE FROM payments WHERE invoice_id IN (SELECT id FROM invoices
          WHERE contract_id IN (...))
        - DELETE FROM invoices WHERE contract_id IN (...)
        - DELETE FROM contracts WHERE tenant_id IN (...)
        - DELETE FROM maintenance_tickets WHERE user_id = ?
        - DELETE FROM expenses WHERE user_id = ?
        - DELETE FROM tenants WHERE user_id = ?
        - DELETE FROM units WHERE property_id IN (SELECT id FROM properties
          WHERE user_id = ?)
        - DELETE FROM properties WHERE user_id = ?
        - DELETE FROM settings WHERE user_id = ?
     b. Insert all data from the backup, preserving original IDs (or
        re-mapping if IDs conflict – in a single-user system, IDs should
        be clean).
     c. Commit transaction.
  10. Backend returns a success response.
  11. React shows a success toast: "✓ تم استعادة البيانات بنجاح. سيتم
      إعادة تحميل الصفحة." (Data restored successfully. The page will
      reload.)
  12. After 2 seconds, the page reloads (window.location.reload()) and
      the landlord sees the restored data on the Dashboard.

Alternative Flows:
  - AF-01: Invalid Backup File – Backend returns HTTP 422:
    "ملف النسخة الاحتياطية غير صالح أو تالف." (The backup file is
    invalid or corrupted.)
  - AF-02: Wrong User Backup – If the backup was exported from a
    different user account, the import fails on user_id mismatch. Message:
    "ملف النسخة الاحتياطية لا ينتمي إلى هذا الحساب." (The backup file
    does not belong to this account.)
  - AF-03: Old Version Backup – If the backup version is 1.x and the
    system is 2.0, return: "النسخة الاحتياطية من إصدار قديم. يرجى
    التحديث." (Backup is from an old version. Please update.)
  - AF-04: Large File – If the file exceeds 50MB, return HTTP 413:
    "حجم الملف كبير جداً. الحد الأقصى 50 ميجابايت." (File too large.
    Maximum 50 MB.)
  - AF-05: Partial Backup Failure – If the database insert fails mid-way,
    the transaction rolls back completely, leaving the original data
    intact. Return HTTP 500 with: "فشلت عملية الاستعادة. لم يتم تغيير
    أي بيانات." (Restore failed. No data was changed.)

Post-conditions:
  - (Backup) A .json file is downloaded to the landlord's device.
  - (Restore) All database records for this user are replaced with the
    backup data.
  - (Restore) The page reloads to reflect restored data.
  - Audit log entry: 'EXPORT_BACKUP' or 'IMPORT_BACKUP'.

Validation Rules:
  - File: Required; .json extension; valid JSON; max 50MB.
  - Structure: Must contain version, data.properties, data.units, etc.
  - User ID: All records must belong to the authenticated user.

  - Settings section: "النسخ الاحتياطي"
  - Button: "نسخ احتياطي للبيانات"
  - Button: "استعادة البيانات من نسخة احتياطية"
  - Success toast (backup): "✓ تم إنشاء النسخ الاحتياطي وتحميله."
  - Success toast (restore): "✓ تم استعادة البيانات بنجاح. سيتم إعادة
    تحميل الصفحة."
  - Warning dialog: "⚠ تحذير: سيتم استبدال جميع البيانات الحالية
    بالبيانات الموجودة في النسخة الاحتياطية. لا يمكن التراجع عن هذا

  - Confirm button: "نعم، استعادة"

3. NON-FUNCTIONAL REQUIREMENTS
================================================================================

3.1 SECURITY REQUIREMENTS
--------------------------
NFR-01: Authentication
  - All API endpoints except /api/auth/login and /api/auth/register (if
    implemented) must require a valid JWT token in the Authorization header.
  - JWT tokens must expire after 7 days.
  - JWT secret must be at least 256 bits and stored in an environment
    variable (JWT_SECRET), not in the codebase.
  - Passwords must be hashed with bcryptjs with a salt round of 12.

NFR-02: Authorization
  - Every database query that involves user-specific data must include
    WHERE user_id = ? (or a JOIN that restricts by user_id) to ensure
    data isolation.
  - Only the landlord who owns a record can read, update, or delete it.
  - The userId is extracted from the JWT token payload, never from the
    request body.

NFR-03: Input Validation & Sanitization
  - All user input must be validated and sanitized on the server side
    using express-validator.
  - HTML tags and script injections must be stripped from all text inputs
    (using trim and escape sanitizers).
  - File uploads must be validated for type, size, and content signature
    (magic bytes check for images).
  - SQL injection is prevented by using parameterized queries (better-
    sqlite3 prepared statements). No raw string concatenation for SQL.

NFR-04: File Upload Security
  - Uploaded files must be stored outside the web root or in a directory
    that is not directly accessible via URL. Files are served only through
    an API endpoint that validates authentication and authorization.
  - Uploaded file names must be sanitized (replace special characters with
    underscores).
  - Execute permission must NOT be set on uploaded files.
  - The uploads directory must be excluded from any version control.

NFR-05: Session Management
  - The JWT token is stored in localStorage (not cookies) to avoid CSRF
    attacks. The trade-off (XSS vulnerability) is mitigated by strict
    Content Security Policy headers.
  - An Axios interceptor must catch HTTP 401 responses globally and
    redirect to the login screen.
  - No session data is stored on the server; the system is stateless.

NFR-06: Audit Logging
  - The following actions must be logged to the audit_log table:
    LOGIN, LOGOUT, ADD_PROPERTY, UPDATE_PROPERTY, DELETE_PROPERTY,
    ADD_UNIT, UPDATE_UNIT, DELETE_UNIT, ADD_TENANT, UPDATE_TENANT,
    DELETE_TENANT, ADD_CONTRACT, TERMINATE_CONTRACT, ADD_PAYMENT,
    ADD_EXPENSE, UPDATE_EXPENSE, DELETE_EXPENSE, ADD_MAINTENANCE_TICKET,
    UPDATE_TICKET_STATUS, EXPORT_BACKUP, IMPORT_BACKUP,
    BULK_INVOICE_GENERATION, SETTINGS_CHANGE, PIN_CHANGE, FAILED_LOGIN.
  - Each log entry includes: user_id, action, target_id (nullable),
    ip_address, user_agent, created_at.

NFR-07: HTTPS
  - In production, the application must be served exclusively over HTTPS.
  - All API responses must include security headers:
    X-Content-Type-Options: nosniff
    X-Frame-Options: DENY
    Content-Security-Policy: default-src 'self'
    Strict-Transport-Security: max-age=31536000; includeSubDomains

3.2 DATA INTEGRITY & VALIDATION
--------------------------------
NFR-08: Database Constraints
  - All foreign key relationships must be enforced at the database level
    using PRAGMA foreign_keys = ON.
  - Required fields must have NOT NULL constraints.
  - Enum fields (status, type, urgency, payment_method) must be validated
    at both the application layer and via CHECK constraints where
    supported by SQLite.

NFR-09: Transactional Integrity
  - All operations that modify multiple tables (e.g., registering a tenant
    updates tenants, units, and contracts) must use database transactions.
  - If any statement in the transaction fails, all changes are rolled back.
  - The transaction isolation level must be SERIALIZABLE (default for
    better-sqlite3 with WAL mode).

NFR-10: Cron Job Idempotency
  - The invoice generation cron job must check for existing invoices
    before creating new ones to prevent duplicate invoices.
  - The cron job must log its start time, end time, and number of invoices
    generated to a cron_log table for monitoring.

NFR-11: Data Consistency on Backup Restore
  - The backup import must validate that all foreign key references within
    the backup are valid (e.g., every unit_id references an existing
    property_id within the backup data).
  - If the backup validation fails, the entire restore transaction is
    rolled back and the original data remains intact.

3.3 PERFORMANCE & AVAILABILITY
-------------------------------
NFR-12: Response Time
  - API responses for read operations (Dashboard, list screens) must
    complete within 500ms for datasets up to 10,000 records.
  - API responses for write operations (create, update, delete) must
    complete within 1 second.
  - File upload responses must complete within 5 seconds for files up
    to 5MB.

NFR-13: Concurrency
  - SQLite with WAL mode supports concurrent reads and a single writer.
    For a single-user system, this is sufficient.
  - If the cron job is writing invoices while the landlord is viewing
    the invoice list, the read will see the committed state (no dirty
    reads).

NFR-14: Uptime
  - The application should have 99.5% uptime during business hours
    (8:00 AM – 10:00 PM, Saudi Arabia time).
  - The cron job must survive server restarts (registered on app startup).

NFR-15: Scalability (Future-Proofing)
  - The database schema and API design must support future multi-user
    expansion. Every table includes a user_id foreign key.
  - The API should use pagination (page and limit query parameters) for
    list endpoints from the start, even if Phase 2 does not use it.
  - The React frontend should be built in a modular component structure
    that allows future features to be added without refactoring existing
    code.

3.4 MOBILE-FIRST RESPONSIVENESS
--------------------------------
NFR-16: Device Support
  - The application must render correctly on screens from 360px width
    (small smartphones) up to 1920px width (desktop monitors).
  - The primary target device is a 10-inch tablet in landscape or portrait
    mode (1024x768 or 1280x800).
  - All touch targets must be at least 48x48 CSS pixels (WCAG 2.1
    guideline).

NFR-17: Layout Behavior
  - On screens narrower than 768px: single-column layout, stacked cards,
    full-width buttons.
  - On screens 768px and wider: two-column grid for summary cards,
    side-by-side form fields where appropriate.
  - Navigation: A persistent bottom navigation bar (4-5 icons with Arabic
    labels) on mobile/tablet. On desktop, a collapsible left sidebar.
  - No horizontal scrolling. All content must fit within the viewport
    width.
  - Modals and dialogs must be centered and not exceed 90% of viewport
    width.

NFR-18: Touch Optimization
  - All interactive elements must have a hover state (for desktop) and
    active state (for touch).
  - Swipe gestures must NOT be used for any action (elderly users may
    not discover or reliably perform swipes).
  - All actions must be accomplished via visible button taps.
  - The back button (browser or in-app) must navigate to the previous
    screen without data loss.

3.5 SENIOR-FRIENDLY UX STANDARDS
---------------------------------
NFR-19: Font and Typography
  - Primary font: "Tajawal" (sans-serif, designed for Arabic text).
  - Fallback font: "Cairo" (also Arabic-friendly).
  - Minimum font size: 16px (for body text).
  - Default font size: 24px (for Extra Large mode).
  - Headings: 28px – 48px depending on hierarchy.
  - Numbers in financial displays: 36px – 48px bold.
  - Font weight: Normal (400) for body, Bold (700) for headings and
    numbers.
  - Line height: 1.6 for readability.

NFR-20: Color Palette
  - Primary color: #1A2E40 (dark blue) – used for buttons, headers,
    navigation.
  - Primary hover: #243B52 (slightly lighter blue).
  - Success color: #2ECC71 (green) – used for paid status, success
    messages, occupied units.
  - Danger color: #E74C3C (red) – used for overdue status, delete
    buttons, errors, vacant units (if > 0).
  - Warning color: #F39C12 (amber) – used for partial payment, medium
    urgency.
  - Background color: #F8F9FA (very light gray) – main page background.
  - Card background: #FFFFFF (pure white).
  - Text color: #1A2E40 (dark blue-black) for high contrast against white
    backgrounds.
  - Muted text: #7F8C8D (gray) for secondary labels.
  - Focus / outline: #3498DB (bright blue) – 3px solid outline on focused
    elements for keyboard navigation.
  - Color contrast ratio: All text/background combinations must meet WCAG
    AAA (7:1) for body text and AA (4.5:1) for large text.

NFR-21: Spacing and Layout
  - Minimum padding around touch targets: 12px.
  - Card padding: 20px.
  - Section spacing: 32px between sections.
  - Border radius: 12px for cards, 8px for buttons, 4px for inputs.
  - Box shadow for cards: 0 2px 8px rgba(0,0,0,0.08) – subtle.

NFR-22: Visual Clarity
  - Icons: Use simple, filled, high-contrast icons (Material Symbols or
    Font Awesome, filled style) alongside every label.
  - Buttons: Must have a clear label + icon. No icon-only buttons.
  - Forms: Labels must be above the input field (not placeholder-only).
    Placeholder text is supplementary.
  - Error messages: Displayed directly below the relevant field in red
    (#E74C3C) with an error icon.
  - Success messages: Green (#2ECC71) toast at the top of the screen,
    auto-dismiss after 3 seconds.
  - Loading states: Pulsing skeleton placeholders, not spinning spinners
    (skeletons are less disorienting for elderly users).
  - Empty states: Illustrative empty-state component with a friendly
    message and a clear action button.

NFR-23: Cognitive Load Reduction
  - Maximum 5 interactive elements per screen (excluding list items).
  - No hamburger menus or hidden navigation.
  - All navigation uses a persistent bottom bar (or sidebar) with large
    labels.
  - Every destructive action requires a confirmation dialog with the
    action clearly described in Arabic.
  - The system must never show technical error codes (e.g., "500 Internal
    Server Error") to the user. Always show a human-readable message in
    Arabic.
  - A consistent "Back" (رجوع) button in the top-left of every detail
    screen.

NFR-24: Accessibility
  - All form inputs must have associated <label> elements (not just
    placeholders).
  - All images must have alt text in Arabic.
  - The application must be navigable via keyboard (Tab, Enter, Escape)
    for users who prefer a keyboard.
  - Focus indicators must be visible (3px solid #3498DB outline).
  - Color must never be the sole means of conveying information (e.g.,
    status badges must have both color AND text).

3.6 ARABIC LOCALIZATION STANDARDS
----------------------------------
NFR-25: RTL Layout
  - The entire application must use dir="rtl" on the <html> element.
  - All CSS margins, paddings, and positioning must use logical
    properties (margin-inline-start, padding-inline-end, etc.) or be
    explicitly mirrored for RTL.
  - Text alignment: Body text right-aligned; numbers and currency left-
    aligned within their containers.
  - Icons that imply direction (e.g., a left arrow for "back") must be
    flipped for RTL (use the appropriate RTL variant or rotate 180°).

NFR-26: Number and Date Formatting
  - All dates must be displayed in Arabic names:
    "الأحد، ٦ يونيو ٢٠٢٦" (Sunday, 6 June 2026).
  - The month names in Arabic: يناير, فبراير, مارس, أبريل, مايو, يونيو,
    يوليو, أغسطس, سبتمبر, أكتوبر, نوفمبر, ديسمبر.
  - Weekday names in Arabic: الأحد, الاثنين, الثلاثاء, الأربعاء, الخميس,
    الجمعة, السبت.
  - Currency formatting: Use Arabic-Indic numerals (٠١٢٣٤٥٦٧٨٩) if the
    user's device locale supports it. Otherwise, use Western digits with
    Arabic currency suffix.
  - Thousands separator: comma (,) for Western digits, or leave as
    continuous for Arabic-Indic.
  - Example: "٢٬٠٠٠ ر.س" or "2,000 ر.س".

NFR-27: Language Consistency
  - The user interface must be 100% in Arabic. No English text should
    appear in the UI except in the following cases:
    - The application's name/logo (if in English, "PMS" is acceptable).
    - Technical terms that have no common Arabic equivalent (rare).
    - URL paths (the user should not see URLs).
  - All error messages returned by the backend must be in Arabic.
  - The server-side validation error messages must be in Arabic.
  - The cron job logs (server-side only) can be in English.

6. CRON JOB SPECIFICATIONS
================================================================================

Job Name: generate_monthly_invoices
Schedule: Daily at 00:00 server time (node-cron: '0 0 * * *')
Description: Scans all active contracts and generates invoices for the
current month if they do not already exist.

Implementation:
  1. On application startup, register the cron job in app.js or server.js:
     const cron = require('node-cron');
     const { generateInvoices } = require('./jobs/invoiceGenerator');
     cron.schedule('0 0 * * *', () => {
       console.log(`[CRON] Starting invoice generation at
         ${new Date().toISOString()}`);
       generateInvoices();
     });
  2. The generateInvoices function:
     function generateInvoices() {
       const startTime = new Date().toISOString();
       const cronLogId = db.prepare(
         'INSERT INTO cron_log (job_name, started_at, status) VALUES (?, ?, ?)'
       ).run('generate_invoices', startTime, 'running').lastInsertRowid;
       try {
         const now = new Date();
         const month = now.getMonth() + 1; // JS months are 0-indexed
         const year = now.getFullYear();
         const settings = getSettingsForAllUsers();
         // Group users by their billingDay setting
         // But since the cron runs daily, check: is today the configured
         // billing day for this user?
         // Actually, simpler approach: run daily and check if an invoice
         // already exists for the current month. If not, generate it.
         // This ensures invoices are generated regardless of billing day
         // setting on the 1st run after the month starts.
         // Refined approach: generate invoices on the 1st of each month.
         // If billingDay is configured differently, generate on that day
         // instead. For Phase 2 MVP: generate on the configured billingDay
         // OR on the 1st if not set.
         const users = db.prepare('SELECT id FROM users WHERE
           is_active = 1').all();
         let totalGenerated = 0;
         for (const user of users) {
           const billingDay = getSetting(user.id, 'billingDay') || '1';
           const billingDayNum = parseInt(billingDay, 10);
           const currentDay = now.getDate();
           // Only generate if today matches billing day OR if we passed
           // the billing day and no invoices exist for this month
           // (catch-up logic)
           if (currentDay >= billingDayNum) {
             const count = generateInvoicesForUser(user.id, month, year);
             totalGenerated += count;
           }
         }
         const endTime = new Date().toISOString();
         db.prepare(
           'UPDATE cron_log SET completed_at = ?, status = ?,
            records_affected = ? WHERE id = ?'
         ).run(endTime, 'completed', totalGenerated, cronLogId);
         console.log(`[CRON] Generated ${totalGenerated} invoices.
           Completed at ${endTime}`);
       } catch (error) {
         const endTime = new Date().toISOString();
         db.prepare(
           'UPDATE cron_log SET completed_at = ?, status = ?,
            error_message = ? WHERE id = ?'
         ).run(endTime, 'failed', error.message, cronLogId);
         console.error(`[CRON ERROR] Invoice generation failed:`,
           error.message);
       }
     }
  3. The function must be idempotent: it checks for existing invoices
     before inserting (SELECT count WHERE contract_id AND billing_month
     AND billing_year). If count > 0, skip.
  4. Logging: Every run is logged in the cron_log table with start time,
     end time, status, and number of invoices generated.
  5. Error Handling: If the job throws an exception, it is caught, logged,
     and the job completes with status 'failed'. The error does NOT crash
     the server. The job will retry on the next scheduled run.
  6. Manual Trigger: An API endpoint POST /api/invoices/trigger is
     available for developers to manually trigger the job for testing
     or recovery.

Invoice Number Generation:
  - Format: INV-{YYYY}-{MM}-{SSS}
  - YYYY: 4-digit year.
  - MM: 2-digit month (zero-padded).
  - SSS: 3-digit sequence number (zero-padded) that resets each month.
  - Example: INV-2026-06-001, INV-2026-06-002, ...
  - The sequence is calculated as:
    SELECT COUNT(*) + 1 FROM invoices WHERE billing_month = ? AND
    billing_year = ?
